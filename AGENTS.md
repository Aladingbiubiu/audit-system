# AGENTS.md

本文件用于约束在本仓库中工作的 Codex / AI 开发代理行为。新会话进入本项目后，应优先阅读并遵守本文件。

## 1. 先读什么

进入项目后，默认按以下顺序获取上下文：

1. `README.md`
2. `handoff.md`
3. `project.md`
4. `config/rules.yaml`
5. `core/rule_engine.py`
6. `core/workflow.py`
7. `core/auditor.py`

不要在未阅读这些文件前，随意重构审核逻辑或恢复旧规则。

## 2. 项目定位

这是一个“行政审批材料审核系统”，当前采用：

- `规则引擎优先`
- `LLM 辅助`
- `OCR 支撑扫描件识别`

目标不是把所有判断都交给大模型，而是：

1. 先用程序处理可确定、可复现、可解释的规则
2. 仅将语义理解、上下文判断类问题交给 LLM
3. 持续减少误判、减少提示词堆砌、提升稳定性

## 3. 开发总原则

### 3.1 规则优先，模型靠后

如果一个判断可以通过以下方式稳定完成，应优先写进 `core/rule_engine.py`，不要优先改 prompt：

- 关键词命中
- 标题识别
- 页码定位
- 固定格式识别
- 数值抽取与比对
- 同类字段一致性校验
- 条件豁免判断
- 明确的误判过滤

只有在“需要理解语义、口径、上下文关系”时，才优先修改 `core/auditor.py` 中的 LLM 流程。

新增确定性规则时，不要继续把所有判断直接写进 `evaluate` 主流程。优先遵循以下结构：

1. 在事实提取层补充 `DocumentFacts`
2. 在团组画像中补充 `GroupProfile` 或相关策略字段
3. 通过 `RuleMetadata` / `Rule` 接入规则
4. 在 `PolicySelector` 对应团组策略中启用规则 ID
5. 仅将复杂抽取工具函数保留在 `core/rule_engine.py`

这样可以避免规则隐形耦合，并让每条规则具备 ID、层级、适用范围和依赖事实。

### 3.2 不要轻易恢复已关闭规则

以下内容目前明确不自动审核，除非用户再次明确要求恢复：

- 呈报表签发人相关全部判断
- 呈报表文号判断
- 预算审批意见表中的 `团长（级别）`
- 落款、盖章、印章、手写区域附近的模糊日期判断

若这些规则再次出现在结果中，优先检查：

1. `config/rules.yaml` 是否被重新写回
2. `core/rule_engine.py` 是否缺少过滤
3. `core/auditor.py` 的后处理是否漏拦截

### 3.3 不要让 summary 跑在 issues 前面

最终页面展示应以“最终问题列表”为准。

如果 `summary` 与 `issues` 冲突，应优先保证：

- `issues` 是最终事实来源
- `summary` 不能提及已被过滤的问题
- 如果需要，重写 `summary`，不要保留模型初稿中的错误描述

### 3.4 优先减少误报

本项目当前阶段的核心目标之一，是降低误报，而不是一味增加“看起来更严格”的检查项。

在新增规则前，先判断它是否满足：

- 可稳定识别
- 可解释
- 页码或证据可回溯
- 不会因为 OCR 抖动大面积误报

如果不满足，倾向：

- 改为提示
- 改为人工判断
- 暂不纳入自动审核

## 4. 当前稳定口径

### 4.1 禁用词

- `参观 / 考察 / 调研`：按严重问题处理
- `学习`：
  - 在 `培训 / 访学 / 学术交流` 场景放过
  - 其他场景可作为禁用词
- 邀请函译文中的禁用词不判断
- 同类禁用词只报一条，不重复列多条

### 4.2 企业团组

团组类型必须以“组团单位”字段值为准，不得因为全文其他位置出现“公司”等词就判为企业团组。例如行程中的“航空公司”、邀请函正文中的其他机构名称，不能作为组团单位类型依据。

若“组团单位”字段值出现以下关键词，优先按企业团组处理：

- `集团`
- `公司`
- `有限公司`
- `股份公司`

企业团组不直接因缺少以下内容判错：

- 列入计划情况
- 周末公务情况
- 翻译情况
- 是否学术交流团

但仍应关注公示情况。

### 4.3 邀请函翻译件

邀请函翻译件审核重点只看：

- 行文逻辑
- 错别字
- 邀请单位中文名称是否一致
- 邀请人是否为外国人 / 外国籍华人

不要因以下细节轻易判“翻译不完整”：

- 地址
- 电邮
- 职务
- 其他次要格式细节

### 4.4 一致性校验

以下一致性校验优先程序化：

- 跨材料停留天数一致性
- 邀请单位中文名称一致性
- 同类字段的一致性比对

不要做以下错误比对：

- 身份证号 vs 护照号
- 中文名称一致但英文大小写 / 空格 / 缩写不同就判错

### 4.5 已知应过滤的误判类型

以下类型如果由 LLM 提出，优先考虑直接过滤：

- OCR 导致的表头重复，如“组团单位重复填写”
- 团组人数与人员名单一致，却说“格式不规范”
- 日期区间与停留天数一致，却说“不符”或“表述不规范”
- OCR 无法稳定识别的盖章 / 手写日期类问题
- 已识别人员名单却说“未附人员名单”
- 已识别交通班次却说“缺少航班号 / 车次”

### 4.6 材料顺序与附件识别范围

默认材料顺序通常按以下顺序理解：

1. `呈报表`
2. `人员名单`
3. `日程安排`
4. `预算表`
5. `情况说明`

对访学团组尤其要注意：

- 后附材料可能很多
- 对后附委派材料、通知、批复、证明类附件，默认只识别第一页
- 不要对这类附件做通篇逐页扫描
- 不要拿附件正文去做禁用词、细碎语义或长文本格式判断

若需识别委派材料，优先只看附件第一页标题和首页关键信息。

## 5. 代码修改建议

### 5.1 改规则时优先检查的文件

- `config/rules.yaml`
- `core/rule_engine.py`
- `core/workflow.py`
- `core/auditor.py`

新增规则时，优先考虑以下流程：

1. 能否由 `rule_engine` 确定性完成
2. 若不能，是否可通过“先提取事实，再交给 LLM”完成
3. 最后才考虑单纯增强 prompt

### 5.2 修改顺序建议

处理审核误判时，优先顺序如下：

1. 先定位是 OCR 问题、规则问题，还是 LLM 问题
2. 若是固定模式误判，先补 `rule_engine` 或过滤逻辑
3. 若是语义问题，再补 prompt 或摘要清洗
4. 修改后尽量用已知材料回放验证

### 5.3 不做无关重构

除非为完成当前任务所必需，不要：

- 大规模改目录结构
- 批量改命名风格
- 引入新的复杂框架
- 重写现有工作流

本项目目前更需要“稳步收敛规则”，而不是大换血。

## 6. 输出与交付要求

在完成修改后，优先说明：

1. 改了哪些文件
2. 为什么这样改
3. 哪些规则被新增、收紧、放宽或关闭
4. 是否做了验证，验证结论是什么

如果某条规则仍不稳定，要明确说“不建议自动化判断”，不要硬凑成自动审核能力。

## 7. 新会话默认目标

若用户没有给出更具体目标，默认优先做这些事：

1. 优化 `core/rule_engine.py` 中可程序判定的规则
2. 降低已知误报
3. 提升 `summary` 与最终 `issues` 的一致性
4. 避免把已经关闭的规则重新带回来

## 8. 维护要求

当项目口径发生明显变化时，应同步更新：

- `project.md`：全局背景
- `handoff.md`：近期交接重点
- `AGENTS.md`：长期自动开发规范

如果只是短期试验，不要急着写进 `AGENTS.md`；只有“已经确认要长期遵守”的规则，才适合固化到这里。

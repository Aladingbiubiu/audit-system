# 行政审批材料智能审核系统说明书

## 1. 系统定位

本系统面向行政审批材料审核场景，重点解决出访任务申报材料中材料完整性、字段一致性、规则合规性和语义合理性等审核问题。

系统采用“规则引擎优先 + 大语言模型辅助”的混合审核架构。对于可程序化、可复现、可解释的审核事项，优先通过规则引擎完成；对于需要上下文理解、行文逻辑判断、语义辨析的事项，再交由大语言模型辅助审核。

该设计目标不是将所有判断交给大语言模型，而是通过结构化规则体系降低误报、提升审核稳定性，并保留模型在复杂语义理解方面的辅助能力。

## 2. 当前问题与优化方向

随着审核规则逐步增加，系统容易出现规则逻辑散落在代码中、规则之间隐形耦合、规则适用范围不清晰等问题。例如，企业团组、高校访学团组、普通机关事业单位团组适用的审核口径并不完全相同，如果在各个函数中零散加入判断条件，后续维护难度会持续增加。

为解决上述问题，系统后续拟建设结构化规则体系，使规则具备“可配置、可解释、可扩展、可测试”的能力，并引入“团组类型驱动审核策略”：

1. 先识别团组类型和任务属性。
2. 根据团组类型选择适用的审核规则集合。
3. 再执行结构规则、一致性规则和语义规则。
4. 最终将程序规则结果与大语言模型结果合并，并以程序事实为准进行过滤和解释。

## 3. 规则分层模型

系统拟将审核规则划分为三类：结构规则、一致性规则、语义规则。

### 3.1 结构规则

结构规则用于判断材料是否具备必要组成部分、字段、附件或明确格式要素。此类规则通常可以通过关键字、标题、页码、字段名、表格内容等方式稳定识别。

典型示例包括：

- 是否附有人员名单。
- 是否识别到呈报表、日程安排、预算审批意见表、情况说明等材料。
- 是否出现航班号、车次等交通班次信息。
- 是否存在公示情况、翻译情况等固定表述。
- 是否出现禁用词，如“调研”“参观”“考察”等。

结构规则应优先由程序规则引擎完成，不应依赖大语言模型自由判断。

### 3.2 一致性规则

一致性规则用于比对不同材料之间的同类字段是否一致。此类规则的关键是先提取事实，再进行字段级比对，避免错误地将不同类型字段混合比较。

典型示例包括：

- 呈报表、邀请函翻译件、预算审批意见表中的停留天数是否一致。
- 呈报表与邀请函译文中的中文邀请单位名称是否一致。
- 团组人数与人员名单人数是否一致。
- 高校访学团组的委派来源与费用来源开支项目是否一致。

一致性规则应遵循“同类字段才可比对”的原则。例如，不应将身份证号码与护照号码互相比对，也不应因为英文机构名称大小写、空格或缩写差异而判定中文邀请单位不一致。

### 3.3 语义规则

语义规则用于处理需要上下文理解、行文逻辑判断或政策口径辨析的问题。此类规则通常不适合完全依靠固定正则或关键字判断。

典型示例包括：

- 邀请函译文行文逻辑是否规范。
- 邀请人身份是否合理，是否为外国人或具有外国国籍的华人姓名。
- 材料说明是否能够支持实际出访任务性质。
- 某些表述是否存在明显错别字、漏字或语义矛盾。

语义规则可由大语言模型辅助完成，但模型输出需经过程序事实和审核策略过滤，避免与已识别事实冲突。

## 4. 事实提取层

为降低规则之间的耦合，系统应先建立统一的事实提取层。事实提取层只负责从材料中提取客观事实，不直接判定是否违规。

可提取的事实包括：

- 团组单位名称。
- 团组类型，如企业团组、高校科研院所团组、普通机关事业单位团组。
- 任务类型，如访学、学术交流、校际交流、一般公务出访。
- 是否附有人员名单。
- 是否识别到公示情况。
- 是否识别到翻译情况。
- 是否识别到交通班次信息。
- 各材料中的停留天数。
- 各材料中的中文邀请单位名称。
- 是否存在国家留学基金委、省派、校派等委派来源。
- 是否识别到相应委派材料。
- 呈报表中的费用来源开支项目。

事实提取层输出应形成统一的数据结构，供后续策略选择和规则判定使用。

## 5. 团组类型驱动审核策略

不同团组类型适用的审核口径不同，因此系统应先识别团组类型，再决定启用哪些规则。

### 5.1 企业团组

当组团单位名称中出现“集团”“公司”“有限公司”“股份公司”等企业特征字样时，可优先按企业团组处理。

企业团组可不机械要求以下内容：

- 列入计划情况。
- 周末公务情况。
- 翻译情况。
- 是否学术交流团。

但企业团组仍应关注公示情况、邀请单位合规性、出访任务合理性等事项。

### 5.2 高校科研院所团组

当组团单位名称中出现“大学”“学院”“学校”“研究院”“研究所”“实验室”等特征字样时，可优先按高校科研院所团组处理。

高校科研院所团组应进一步识别任务类型：

- 访学。
- 学术交流。
- 校际交流。

对于长期访学团组，重点关注公示情况、委派材料和经费来源一致性；如呈报表末尾已明确写明公示情况，不应机械要求列入计划情况、周末公务情况、翻译情况等表述。

对于学术交流或校际交流团组，应重点关注呈报表最后一页是否标注类似“此团系学术交流团组”的说明。

### 5.3 普通机关事业单位团组

普通机关事业单位团组可执行常规审核口径，重点关注呈报表结尾字段、人员名单、行程安排、预算审批意见表、邀请函合规性等内容。

## 6. 规则元数据层设计

为使规则可解释、可扩展，系统已开始引入规则元数据层。每条规则不再只是散落在代码中的条件判断，而是具有明确身份、适用范围和说明的规则对象。

规则元数据可包含以下字段：

- `id`：规则唯一标识。
- `name`：规则名称。
- `layer`：规则层级，如结构规则、一致性规则、语义规则。
- `severity`：默认严重程度，如严重、一般、提示。
- `applies_to`：适用团组类型。
- `requires_facts`：规则依赖的事实字段。
- `decision`：规则判定函数。
- `explanation_template`：问题解释模板。
- `enabled`：是否启用。
- `version`：规则版本。

示例：

```python
Rule(
    id="duration.cross_material_consistency",
    name="跨材料停留天数一致性",
    layer="consistency",
    severity="严重",
    applies_to=["enterprise", "academic", "government"],
    requires_facts=["duration_values"],
    decision=check_duration_consistency,
    explanation_template="停留时间跨材料不一致：{details}",
)
```

该设计保留 Python 函数处理复杂逻辑，同时通过元数据实现规则可识别、可开关、可解释和可测试。

## 7. 推荐审核流程

系统后续推荐采用如下审核流程：

```text
PDF/OCR 文本
  ↓
DocumentFacts 事实提取
  ↓
GroupProfile 团组类型识别
  ↓
AuditPolicy 审核策略选择
  ↓
RuleRunner 执行结构规则 / 一致性规则
  ↓
LLM 辅助审核语义规则
  ↓
PostFilter 按事实和策略过滤模型误判
  ↓
FinalIssues + Summary 最终问题列表与摘要
```

该流程中，程序事实和规则策略优先级高于大语言模型自由判断。若模型输出与程序事实冲突，应优先以程序事实为准。

## 8. 第一批建议迁移的规则

为降低重构风险，规则结构化不宜一次性完成。建议先选择稳定、可验证的规则作为第一批迁移对象：

- 禁用词规则：属于结构/文字规则。
- 人员名单存在性规则：属于结构规则。
- 跨材料停留天数一致性规则：属于一致性规则。
- 邀请单位中文名称一致性规则：属于一致性规则。
- 企业团组结尾字段豁免：属于策略规则，不应散落在普通审核规则中。

第一批迁移目标是建立规则对象、事实模型和策略选择的基本骨架，而不是立刻将所有规则完全配置化。

当前已完成第一版代码骨架：

- `core/rule_model.py` 中已建立 `DocumentFacts`、`GroupProfile`、`AuditPolicy`、`PolicySelector`、`RuleMetadata`、`Rule`、`RuleRunner`。
- `core/rule_engine.py` 中已将主流程拆为事实提取、策略选择、事实展示、策略说明和规则执行。
- 第一批已接入规则元数据执行器的规则包括禁用词审核、跨材料停留天数一致性、邀请单位中文名称一致性和高校科研院所团组策略审核。

## 9. 可解释性与扩展价值

结构化规则体系可以为系统带来以下价值：

- 可解释：每个审核问题都能追溯到具体规则 ID、规则名称和事实依据。
- 可配置：不同团组类型可启用不同规则集合。
- 可扩展：新增规则时只需新增规则对象和判定函数，减少对既有逻辑的影响。
- 可测试：每条规则可独立进行单元测试和样例回放。
- 可审计：误报发生时可判断是事实提取错误、策略选择错误、规则判定错误，还是模型语义判断错误。
- 可维护：避免规则逻辑长期散落在代码中形成隐形耦合。

## 10. 后续建设方向

后续系统可围绕以下方向继续完善：

1. 继续完善 `DocumentFacts`，将更多事实提取结果统一纳入事实层。
2. 继续完善 `GroupProfile`，细化企业团组、高校访学、学术交流、普通机关事业单位团组等画像。
3. 继续细化 `AuditPolicy` 和 `PolicySelector`，后续可将策略启用关系配置化。
4. 将现有确定性规则逐步迁移为带元数据的规则对象。
5. 保留大语言模型作为语义审核辅助，并用规则事实过滤模型误判。
6. 将最终 `summary` 始终基于最终 `issues` 生成，确保摘要与问题列表一致。

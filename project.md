# 项目总上下文

## 项目概况

- 项目路径：`d:\My Project\claude test\audit-system`
- 技术栈：
  - 前端：`Streamlit`
  - 数据：`SQLite + SQLAlchemy`
  - PDF 解析：`pdfplumber`
  - OCR：`PyMuPDF + rapidocr-onnxruntime`
  - 大模型审核：智谱 / Claude
- 启动方式：
  - `python -m streamlit run app.py --server.address 127.0.0.1 --server.port 8501`
- 当前服务地址通常为：
  - `http://127.0.0.1:8501`

## 当前架构

- 已从“纯 LLM 审核”改成“规则引擎优先 + LLM 辅助”的混合架构。
- 关键文件：
  - `app.py`：Streamlit 界面
  - `core/workflow.py`：工作流主入口，先跑规则引擎，再跑 LLM，再合并结果
  - `core/rule_engine.py`：事实提取、规则选择和程序可判定规则入口
  - `core/rule_model.py`：事实模型、团组画像、规则元数据和规则执行器
  - `core/auditor.py`：LLM 审核、结果解析、摘要清洗
  - `core/pdf_parser.py`：PDF 文本 / OCR 提取
  - `core/database.py`：审核记录、反馈判例存储
  - `core/learning.py`：口径 / 反馈学习
  - `config/rules.yaml`：当前审核规则
  - `config/guidelines.yaml`：当前基本空置，可保留为 `guidelines: []`

## 已完成的关键能力

### 1. OCR

- 扫描页会自动 OCR。
- 已能识别扫描型呈报表、人员名单等内容。

### 2. 规则引擎优先

- 已新增结构化规则骨架：
  - `DocumentFacts`：统一承载材料事实
  - `PresentmentFacts`：统一承载 PDF/OCR 呈报表字段事实，区分组团/随团表
  - `BudgetFacts`：统一承载 PDF/OCR 预算审批意见表字段事实
  - `GroupProfile`：统一描述团组类型和任务类型
  - `AuditPolicy` / `PolicySelector`：按团组类型显式选择启用的规则 ID
  - `RuleMetadata` / `Rule`：为规则提供 ID、层级、适用范围和依赖事实
  - `RuleRunner`：按团组策略执行已启用规则
- 程序先识别这些事实：
  - 企业团组
  - 人员名单
  - 公示情况
  - 翻译情况
  - 交通班次信息
  - 跨材料停留天数
  - 邀请单位中文名称
- 程序可直接生成部分问题，并过滤与程序事实冲突的模型误报。
- 呈报表字段解析已开始结构化，当前可从 PDF/OCR 文本中提取：
  - 呈报表类型：组团 / 随团
  - 组团单位
  - 出访地 / 经停地
  - 停留时间 / 在外停留天数
  - 邀请单位中文名 / 外文名（优先字段锚点）
  - 费用来源 / 开支项目
  - 出访事由（仍需继续增强 OCR 拆字场景）
- 预算审批意见表字段解析已开始结构化，当前可从 PDF/OCR 文本中提取：
  - 单位名称、姓名、职务、团组名称、组团单位、团长（级别）、团组人数
  - 出访国别（地区）（含经停）、出访时间（天数）
  - 审核内容中的“是/否”项
  - 须事先报批的支出事项、其他事项
  - 费用金额明细因 OCR 横向表格易切碎，暂不自动抽取金额
- 第一批已接入元数据执行器的规则包括：
  - 禁用词审核
  - 跨材料停留天数一致性
  - 邀请单位中文名称一致性
  - 高校科研院所团组策略审核

### 3. 结果后处理

- 问题列表排序规则：
  - 严重 -> 一般 -> 提示
  - 同级按类别
  - 再按页码
- 解析器增强：
  - 可去除 ```json 外壳
  - 可修复模型 JSON 中未转义引号导致的坏 JSON
- summary 已改为基于最终 issues 重新生成，避免保留已被过滤的问题描述。

## 当前已程序化 / 已收敛的规则

### 1. 禁用词

- `参观 / 考察 / 调研`：按严重问题处理。
- `学习`：默认算禁用词，但如果出现在 `培训 / 访学 / 学术交流` 场景，则放过。
- 不检查邀请函译文中的禁用词。
- 同类禁用词只合并成一条问题，不重复列多条。

### 2. 企业团组识别

- 团组类型识别已改为只基于“组团单位”字段值判断，不再全文扫描 `公司` 等关键词，避免被“航空公司”等行程文本误伤。
- 组团单位字段值出现 `集团 / 公司 / 有限公司 / 股份公司` 等，优先按企业团组处理。
- 组团单位字段值出现 `大学 / 学院 / 学校 / 研究院 / 研究所 / 实验室` 等，优先按高校科研院所团组处理。
- 企业团组豁免：
  - 不直接因缺少 `列入计划情况 / 周末公务情况 / 翻译情况 / 是否学术交流团` 判错
- 但仍保留对 `公示情况` 的识别逻辑。

### 3. 人员名单识别

- 出现 `团组人员名单 / 人员名单` 标题，或名单式内容（姓名、身份证号、工作单位及职务）即视为已附人员名单。
- 已能过滤“未附人员名单”的误报。

### 3.1 默认材料顺序

- 当前项目中常见材料顺序通常为：
  - `呈报表`
  - `人员名单`
  - `日程安排`
  - `预算表`
  - `情况说明`
- 新规则设计和结构识别时，应优先尊重这一顺序。

### 4. 跨材料停留天数一致性

- 已程序识别并比对：
  - 呈报表：`停留时间 XX 天`
  - 邀请函翻译件：`在国外停留时间 XX 天`
  - 预算审批意见表：`出访时间（天数）XX 天`
- 若不一致，程序直接报：
  - 分类：`跨材料一致性校验`
  - 严重级别

### 5. 邀请单位中文名称一致性

- 已加入高置信句式抽取，优先识别 `应/受 XXX 邀请`、`邀请单位：XXX`、`XXX 诚邀/邀请` 等正文片段。
- 已对联系人、地址、电话、邮箱、附件、承办单位等噪声字段做截断或回退过滤，减少把页眉页脚和联系方式中的机构名误当邀请单位。
- 已做核心中文名称归一化，能将“联邦国家预算高等教育机构 XXX 大学”等外层描述归并到核心机构名。
- 后续仍建议结合真实材料回放继续调参。

### 6. 交通班次信息

- 程序会在行程页识别航班号 / 车次模式（如 `SU205`、`CZ342`、`123 次`）。
- 目前用于过滤模型“缺少航班号 / 车次”的误报。
- 还没有做到逐交通段精确校验，只是“材料中存在交通班次信息”级别。

### 7. 访学团组附件识别范围

- 对访学团组，后附材料可能很多。
- 后附委派材料、通知、批复、证明类附件，默认只识别第一页。
- 不对这类附件通篇逐页扫描。
- 不拿附件正文去做禁用词、细碎语义和长文本格式判断。
- 若需识别国家留学基金委、省派、校派等委派材料，优先只看附件第一页标题和首页关键信息。

## 当前明确不再自动审核 / 已关闭的规则

以下内容已从规则中删除，或在过滤层直接拦掉：

- 呈报表签发人相关：
  - 不再判断“签发人不能为空”
  - 不再判断“签发人不能是出访人本人”
  - 不再判断“签发人必须手写”
- 呈报表文号：
  - 暂时不检测
- 预算审批意见表：
  - `团长（级别）` 这条暂时不检测
- 落款 / 盖章 / 印章附近日期识别不清：
  - 不再当成明确错误，倾向人工核对
- OCR 表头重复：
  - 如“组团单位”重复，已作为 OCR 误判过滤
- 团组人数与名单一致却说“格式不规范”：
  - 已过滤
- 日期区间和天数一致却说“表述不规范”：
  - 已过滤

## 当前规则口径的重要结论

- 邀请函翻译件审核重点只看：
  - 行文逻辑
  - 错别字
  - 邀请单位中文名称是否一致
  - 邀请人是否为外国人 / 外国籍华人
- 不要因地址、电邮、职务等细节轻易判“翻译不完整”。

- 邀请单位名称比对：
  - 以中文为准
  - 不要因英文大小写、空格、缩写差异判错

- 身份证号和护照号：
  - 不做互相比对
  - 不能因此判“不一致”

- 不全程参团停留天数：
  - 只有存在不全程参团人员时才要求在出访事由注明
  - 全员全程参团不作为问题

- `学习`：
  - 在 `培训 / 访学 / 学术交流` 场景可放过
  - 其他场景仍可按禁用词处理

- 邀请函译文中的禁用词：
  - 不提示
  - 不纳入判断

- 禁用词问题：
  - 只指出一次
  - 不重复列多条

- 预算审批意见表中的 `其他费用`：
  - 不要求填“是”
  - 只要有费用明细即可
  - 明细可以写在 `其他费用` 栏，或写在 `其他事项` 中

## 当前存在的问题 / 下一步建议

### 1. 邀请单位中文名称抽取需结合真实材料继续回放

- 基础优化已完成：高置信正文句式、噪声字段截断、核心名称归一化。
- 后续重点是用实际 PDF OCR 结果回放，继续补充稳定句式，避免为少数样例扩大误报面。

### 2. 公示 / 翻译识别还不够稳

- 当前企业团组场景下因为已有过滤，不会误报太多。
- 但事实提取本身仍可更精确。

### 3. 交通信息程序化还可以继续增强

- 目前只是“存在班次信息”级别。
- 可升级为“逐段行程”识别：
  - 日期
  - 出发地 / 到达地
  - 航班号 / 车次
  - 起止时间

### 4. summary 与最终 issues 的强一致性已完成基础收口

- 当前已在最终 issues 完成过滤、追加和排序后，统一基于最终问题列表重写 summary。
- 后续如需优化，可继续改进 summary 的展示文案和分类归纳粒度。

### 5. 更多可程序判定规则仍可继续迁出 LLM

- 当前方向正确，后续继续把确定性规则下沉到 `core/rule_engine.py`。
- 新增确定性规则时，优先通过 `core/rule_model.py` 中的 `RuleMetadata` / `Rule` 接入，并在 `PolicySelector` 对应团组策略中启用规则 ID，而不是继续把判断散落在 `evaluate` 主流程里。

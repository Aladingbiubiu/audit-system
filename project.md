# 项目总上下文

## 项目概况

- 项目路径：`d:\My Project\claude test\audit-system`
- 技术栈：
  - 前端：`Streamlit`
  - 数据：`SQLite + SQLAlchemy`
  - PDF 解析：`pdfplumber`
  - OCR：`PyMuPDF + rapidocr-onnxruntime`
  - 大模型审核：智谱 / Claude
- 启动方式：
  - `python -m streamlit run app.py --server.address 127.0.0.1 --server.port 8501`
- 当前服务地址通常为：
  - `http://127.0.0.1:8501`

## 当前架构

- 已从“纯 LLM 审核”改成“规则引擎优先 + LLM 辅助”的混合架构。
- 关键文件：
  - `app.py`：Streamlit 界面
  - `core/workflow.py`：工作流主入口，先跑规则引擎，再跑 LLM，再合并结果
  - `core/rule_engine.py`：程序可判定规则
  - `core/auditor.py`：LLM 审核、结果解析、摘要清洗
  - `core/pdf_parser.py`：PDF 文本 / OCR 提取
  - `core/database.py`：审核记录、反馈判例存储
  - `core/learning.py`：口径 / 反馈学习
  - `config/rules.yaml`：当前审核规则
  - `config/guidelines.yaml`：当前基本空置，可保留为 `guidelines: []`

## 已完成的关键能力

### 1. OCR

- 扫描页会自动 OCR。
- 已能识别扫描型呈报表、人员名单等内容。

### 2. 规则引擎优先

- 程序先识别这些事实：
  - 企业团组
  - 人员名单
  - 公示情况
  - 翻译情况
  - 交通班次信息
  - 跨材料停留天数
  - 邀请单位中文名称
- 程序可直接生成部分问题，并过滤与程序事实冲突的模型误报。

### 3. 结果后处理

- 问题列表排序规则：
  - 严重 -> 一般 -> 提示
  - 同级按类别
  - 再按页码
- 解析器增强：
  - 可去除 ```json 外壳
  - 可修复模型 JSON 中未转义引号导致的坏 JSON
- summary 会做部分清洗，避免和 issues 明显矛盾。

## 当前已程序化 / 已收敛的规则

### 1. 禁用词

- `参观 / 考察 / 调研`：按严重问题处理。
- `学习`：默认算禁用词，但如果出现在 `培训 / 访学 / 学术交流` 场景，则放过。
- 不检查邀请函译文中的禁用词。
- 同类禁用词只合并成一条问题，不重复列多条。

### 2. 企业团组识别

- 组团单位出现 `集团 / 公司 / 有限公司 / 股份公司` 等，优先按企业团组处理。
- 企业团组豁免：
  - 不直接因缺少 `列入计划情况 / 周末公务情况 / 翻译情况 / 是否学术交流团` 判错
- 但仍保留对 `公示情况` 的识别逻辑。

### 3. 人员名单识别

- 出现 `团组人员名单 / 人员名单` 标题，或名单式内容（姓名、身份证号、工作单位及职务）即视为已附人员名单。
- 已能过滤“未附人员名单”的误报。

### 4. 跨材料停留天数一致性

- 已程序识别并比对：
  - 呈报表：`停留时间 XX 天`
  - 邀请函翻译件：`在国外停留时间 XX 天`
  - 预算审批意见表：`出访时间（天数）XX 天`
- 若不一致，程序直接报：
  - 分类：`跨材料一致性校验`
  - 严重级别

### 5. 邀请单位中文名称一致性

- 已开始程序化，但当前抽取逻辑仍不够稳，尤其中文邀请函页容易抽到抬头或杂质。
- 现状：做了“核心中文名称包含关系”匹配，已比之前好，但仍建议继续优化。

### 6. 交通班次信息

- 程序会在行程页识别航班号 / 车次模式（如 `SU205`、`CZ342`、`123 次`）。
- 目前用于过滤模型“缺少航班号 / 车次”的误报。
- 还没有做到逐交通段精确校验，只是“材料中存在交通班次信息”级别。

## 当前明确不再自动审核 / 已关闭的规则

以下内容已从规则中删除，或在过滤层直接拦掉：

- 呈报表签发人相关：
  - 不再判断“签发人不能为空”
  - 不再判断“签发人不能是出访人本人”
  - 不再判断“签发人必须手写”
- 呈报表文号：
  - 暂时不检测
- 预算审批意见表：
  - `团长（级别）` 这条暂时不检测
- 落款 / 盖章 / 印章附近日期识别不清：
  - 不再当成明确错误，倾向人工核对
- OCR 表头重复：
  - 如“组团单位”重复，已作为 OCR 误判过滤
- 团组人数与名单一致却说“格式不规范”：
  - 已过滤
- 日期区间和天数一致却说“表述不规范”：
  - 已过滤

## 当前规则口径的重要结论

- 邀请函翻译件审核重点只看：
  - 行文逻辑
  - 错别字
  - 邀请单位中文名称是否一致
  - 邀请人是否为外国人 / 外国籍华人
- 不要因地址、电邮、职务等细节轻易判“翻译不完整”。

- 邀请单位名称比对：
  - 以中文为准
  - 不要因英文大小写、空格、缩写差异判错

- 身份证号和护照号：
  - 不做互相比对
  - 不能因此判“不一致”

- 不全程参团停留天数：
  - 只有存在不全程参团人员时才要求在出访事由注明
  - 全员全程参团不作为问题

- `学习`：
  - 在 `培训 / 访学 / 学术交流` 场景可放过
  - 其他场景仍可按禁用词处理

- 邀请函译文中的禁用词：
  - 不提示
  - 不纳入判断

- 禁用词问题：
  - 只指出一次
  - 不重复列多条

- 预算审批意见表中的 `其他费用`：
  - 不要求填“是”
  - 只要有费用明细即可
  - 明细可以写在 `其他费用` 栏，或写在 `其他事项` 中

## 当前存在的问题 / 下一步建议

### 1. 邀请单位中文名称抽取仍需优化

- 尤其中文邀请函页容易混入发函单位抬头、承办单位、学校名等杂项。
- 建议做更稳的正文区抽取，而不是简单 regex。

### 2. 公示 / 翻译识别还不够稳

- 当前企业团组场景下因为已有过滤，不会误报太多。
- 但事实提取本身仍可更精确。

### 3. 交通信息程序化还可以继续增强

- 目前只是“存在班次信息”级别。
- 可升级为“逐段行程”识别：
  - 日期
  - 出发地 / 到达地
  - 航班号 / 车次
  - 起止时间

### 4. summary 与最终 issues 的强一致性还可以继续加强

- 目前已做部分清洗。
- 后续建议改为“summary 基于最终 issues 重写”。

### 5. 更多可程序判定规则仍可继续迁出 LLM

- 当前方向正确，后续继续把确定性规则下沉到 `core/rule_engine.py`。
